# VSCode AI Commit 插件需求文档

## 1. 产品概述

### 1.1 产品背景
Trae 中内置的 AI 一键生成 git commit 信息功能受到用户欢迎，但用户希望在 VSCode 中也能使用类似功能。本插件旨在为 VSCode 用户提供同样强大的 AI 生成 commit 信息功能，支持自定义 AI 大模型。

### 1.2 产品目标
- 提供与 Trae 相同的 AI 生成 commit 信息功能
- 支持自定义 AI 大模型配置
- 无缝集成到 VSCode 的 Git 工作流中
- 提供简洁直观的用户界面
- 确保性能和安全性

## 2. 核心功能

### 2.1 代码变更分析
- 自动分析 git 暂存区的代码变更
- 提取变更的文件、行数和具体内容
- 识别变更类型（新增、修改、删除）
- 当暂存区为空时，提示用户是否需要执行 `git add`，而非静默失败
- 对大型 diff 进行智能截断：按文件分组摘要，或只取每个 hunk 的头部行，确保不超过 Token 预算
- 🆕 **Diff 来源选择**：默认分析暂存区（`git diff --cached`），用户可通过 `aicommit.diffSource` 配置切换为工作区未暂存变更（`git diff`），适用于希望在暂存前预览 commit message 的场景

### 2.2 AI 生成 Commit 信息
- 调用配置的 AI 大模型生成 commit 信息
- 支持多种 commit 格式（Conventional Commits、Gitmoji 等）
- 生成符合规范的 commit 信息，包括类型、范围、描述和详细说明
- **多候选方案**：一次生成 2-3 条候选 commit message，用户可快速切换选择，也可在任意一条基础上编辑。在 prompt 层面要求 AI 返回多个候选，几乎不增加额外成本

### 2.3 预览和编辑
- 生成的 commit 信息**直接填入 VSCode 原生 SCM 面板的 commit 输入框**，用户可在输入框中直接编辑
- 提供快速修改选项（如更改 commit 类型）
- 当存在多候选方案时，在 SCM 输入框上方显示候选切换按钮
- 独立预览窗口作为可选的高级功能（如查看 AI 推荐的多个备选 commit message 的详细对比）

### 2.4 一键提交
- 支持一键提交生成的 commit 信息
- 集成到 VSCode 的 Git 面板中
- 提供命令面板命令和快捷键
- 提供 `AI Commit: 生成并提交` 命令，跳过预览直接提交，适合熟练用户
- 提供对上次生成结果的快速复用（当 diff 未变化时）
- 🆕 **Amend commit 支持**：提供 `AI Commit: 修改上次提交信息` 命令，对最近一次 commit 重新生成 commit message 并执行 `git commit --amend`。适用于刚提交但消息不满意的场景，使用 `HEAD~1` 的 diff 作为输入

### 2.5 边缘场景处理

以下场景需明确检测方式和处理策略：

| 场景 | 检测方式 | 处理策略 |
|------|----------|----------|
| 暂存区为空 | `git diff --cached` 返回空 | 提示用户："暂存区无变更，是否暂存所有变更？"并提供 `git add` 快捷操作 |
| 非 Git 仓库 | `git rev-parse` 返回非零 | 提示用户："当前工作区不是 Git 仓库"，禁用相关命令 |
| 合并冲突未解决 | `git ls-files --unmerged` 非空 | 提示用户："存在未解决的合并冲突，请先解决后再生成" |
| 大仓库/巨型 diff | diff 行数超过阈值（默认 3000 行） | 弹窗提示并让用户选择：截断处理 / 仅分析部分文件 / 取消 |
| AI 返回格式不符 | 解析 commit message 结构失败 | 降级为将 AI 原始输出作为纯文本填入，并提示用户手动调整 |
| API 余额不足/额度耗尽 | HTTP 429 或 402 状态码 | 明确提示"API 额度不足"，与网络错误区分，建议用户检查账户 |
| 网络超时 | 请求超过 30 秒无响应 | 提示"网络超时"，提供重试按钮，不自动重试避免等待堆叠 |
| AI 返回空内容 | 响应 body 为空或 content 为空 | 提示"AI 未返回有效内容"，提供重新生成选项 |
| 🆕 多工作区多仓库 | `workspaceFolders.length > 1` | 命令面板操作时弹出 QuickPick 选择目标仓库 |
| 🆕 Amend 但无上次提交 | `git log` 返回空 | 提示"当前分支无提交历史，无法执行 amend" ✅ |
| 🆕 代理连接失败 | 代理请求超时或 ECONNREFUSED | 提示"代理连接失败，请检查代理配置"，建议检查地址和端口 ✅ |
| 🆕 当前配置组 API 密钥为空 | SecretStorage 中未找到对应 key | 提示"配置组 [组名] 的 API 密钥未设置"，引导用户输入密钥 ✅ |
| 🆕 切换到已删除的配置组 | `activeModelGroup` 指向不存在的组 | 自动回退到第一个可用配置组，并提示"配置组 [组名] 已不存在，已切换到 [回退组]" ✅ |

## 3. 用户界面设计

### 3.1 命令面板
- 提供 `AI Commit: 生成 Commit 信息` 命令
- 提供 `AI Commit: 生成并提交` 命令（跳过预览直接提交）
- 提供 `AI Commit: 重新生成` 命令（复用上次 diff 重新调用 AI）
- 提供 `AI Commit: 配置` 命令
- 提供 `AI Commit: 取消生成` 命令（取消正在进行的 AI 请求）
- 🆕 提供 `AI Commit: 修改上次提交信息` 命令（Amend commit）
- 🆕 提供 `AI Commit: 切换模型配置组` 命令（弹出 QuickPick 选择目标组）

### 3.2 Git 面板集成
- 在 Git 面板的 commit 输入框区域添加 "AI 生成" 按钮（星标/魔法棒图标）
- 点击后：执行 AI 生成，结果直接填入 commit 输入框
- 生成过程中按钮显示 loading 动画，点击可取消
- 提供编辑和提交选项

### 3.3 状态栏
- 🆕 显示当前活跃的模型配置组名称（如 "GPT-4o" 或 "DeepSeek"），点击可快速切换
- 显示插件状态（如 AI 模型连接状态）
- 提供快速访问命令的按钮
- **生成过程反馈**：AI 生成期间，状态栏显示旋转图标 + "AI Commit: 生成中..."，完成后恢复正常状态

### 3.4 配置界面
- **优先使用 VSCode 原生 Settings UI**：通过 `package.json` 的 `contributes.configuration` 声明所有配置项，用户在 VSCode Settings 界面中直接配置。这是最符合 VSCode 用户习惯且开发成本最低的方式
- 仅对需要复杂交互的功能（如"测试连接"按钮、多候选方案预览）才使用 Webview 面板
- 支持 AI 模型选择和 API 密钥配置
- 提供 commit 格式和行为设置

## 4. 配置选项

### 4.1 AI 模型配置
- 支持多种 AI 模型：OpenAI GPT-3.5/4、Anthropic Claude、Google Gemini 等
- **采用 OpenAI 兼容协议作为统一接口**：当前大多数 AI 提供商（包括 DeepSeek、Qwen 等国内厂商）均支持 OpenAI 格式的 API，用户只需配置不同的 `base_url` 和 `model` 即可接入，无需为每个厂商单独适配
- 对不支持 OpenAI 格式的模型（如 Anthropic Claude），提供独立适配层
- 🆕 **AI 模型配置组**：支持配置多组 AI 模型供应，每组包含完整的连接参数，使用时可快速切换
  - **配置组结构**：每组包含 `name`（组名）、`model`（模型名）、`apiKey`（API 密钥）、`baseUrl`（基础 URL）、`temperature`（温度）、`maxTokens`（最大令牌数）
  - **默认配置组**：插件首次使用时创建名为 "Default" 的配置组，兼容单配置的旧版行为
  - **多组管理**：支持增删改配置组，至少保留一组，无法删除当前活跃组
  - **快速切换**：通过命令面板 `AI Commit: 切换模型配置组` 或状态栏点击当前组名，弹出 QuickPick 选择目标组
  - **使用场景**：
    - 工作项目使用公司私有部署模型，个人项目使用 OpenAI
    - 日常使用低成本模型（如 DeepSeek），重要提交使用高质量模型（如 GPT-4o）
    - 主力模型额度耗尽时快速切换到备用模型
  - **配置存储**：配置组列表存储在 `aicommit.modelGroups`（VSCode Settings），API 密钥存储在 SecretStorage（key 格式：`aicommit.apiKey.{groupName}`）
  - **当前活跃组**：通过 `aicommit.activeModelGroup` 记录当前使用的组名，插件启动时自动加载
- 支持模型参数调整（如温度、最大令牌数），每组参数独立
- API 密钥支持从环境变量读取（格式：`${env:API_KEY_NAME}`）

### 4.2 Commit 格式配置
- 支持 Conventional Commits 格式（标准格式，标题与正文间空行分隔）
- 支持 Gitmoji 集成（在 Conventional 基础上增加 Emoji 前缀）
- 支持 Bullet 要点式格式（AI Commit 内置风格，标题末尾加冒号，正文每行一个变更点，`;` 分隔，末行 `。` 结尾）
- 支持自定义 commit 格式模板
- 支持 commit 信息长度限制

### 4.3 行为配置
- 自动检测变更范围
- 保存生成历史
- 生成详细度调整
- 快捷键配置
- 生成候选数量配置（默认 2 条，范围 1-5）
- Diff 截断阈值配置（默认 3000 行）
- 🆕 **项目级配置文件**：支持在项目根目录放置 `.aicommitrc.json`，团队可共享 prompt 模板、格式规范等配置，避免每人手动配置
  - 配置优先级：项目级 `.aicommitrc.json` > 用户级 VSCode Settings > 默认值
  - `.aicommitrc.json` 支持的字段：`promptTemplate`、`commitFormat`、`commitLanguage`、`diffSource`、`candidatesCount`、`maxLength`、`activeModelGroup`（指定项目默认使用的配置组名称）
  - 不含敏感字段（API 密钥等），可安全提交到版本库
  - 配置文件变更后自动重新加载，无需重启 VSCode
  - 🆕 项目级 `activeModelGroup` 可覆盖用户级默认：如工作项目固定使用 "公司私有部署" 组，个人项目使用 "GPT-4o" 组

### 4.4 安全配置
- API 密钥加密存储（使用 VSCode 的 SecretStorage API）
- 支持环境变量配置 API 密钥
- 代码变更数据处理策略

## 5. 技术实现方案

### 5.1 插件架构
- 使用 VSCode 插件 API 开发
- 采用 TypeScript 语言
- 模块化设计，分离核心功能和 UI 组件
- 模块划分：
  - `git/`：Git 操作层（diff 获取、commit 执行）
  - `ai/`：AI 调用层（prompt 构建、API 请求、响应解析）
  - `config/`：配置管理层（设置读写、密钥管理、🆕 配置组管理）
  - `ui/`：界面交互层（命令注册、状态栏、通知）
  - `utils/`：通用工具（日志、错误处理）

### 5.2 Git 集成
- **分层策略**：优先使用 `child_process` 调用 `git` CLI 获取 diff 和执行 commit，这是最稳定可靠的方式，不依赖 VSCode 内部 API
- VSCode 的 `vscode.git` 扩展 API（`getExtension('vscode.git')`）**仅作为辅助**，用于获取 Git 状态信息（如仓库是否初始化、是否有冲突），且需注意该 API 为非公开 API，可能随版本变化
- 处理 git 分支和工作区状态
- 需处理 git 未安装或不在 PATH 中的情况
- 🆕 **多工作区支持**：VSCode 支持多根工作区（Multi-root Workspace），当存在多个 Git 仓库时：
  - 使用 `vscode.workspace.workspaceFolders` 遍历所有工作区目录
  - 在 SCM 面板中，从用户当前聚焦的 SCM 提供者获取对应仓库路径，作为 git 命令的 `cwd`
  - 命令面板操作时，若存在多个仓库，弹出 QuickPick 让用户选择目标仓库
  - 状态栏显示当前活跃仓库的分支信息

### 5.3 AI API 调用
- **采用 OpenAI 兼容协议作为统一调用接口**
  - 默认使用 OpenAI Chat Completions 格式（`POST /v1/chat/completions`）
  - 用户通过配置不同的 `base_url` + `model` 即可接入任何兼容 OpenAI 格式的服务商
  - 对 Anthropic Claude 等不兼容 OpenAI 格式的模型，提供独立适配器
- 🆕 **配置组动态选择**：每次发起 AI 请求时，从当前活跃的模型配置组中读取 `model`、`apiKey`、`baseUrl`、`temperature`、`maxTokens` 等参数，无需重启插件即可切换
  - 切换配置组后，后续请求立即使用新组的参数
  - 配置组信息变更后自动热更新，无需重载窗口
  - API 密钥从 SecretStorage 按 `aicommit.apiKey.{groupName}` key 读取
- 实现请求重试和错误处理
  - 网络错误：最多重试 2 次，间隔 1s/2s 递增
  - 速率限制（429）：提示用户稍后重试，不自动重试
  - 超时：30 秒超时，提供取消操作
- AI 请求为异步操作，支持 CancellationToken 取消
- 🆕 **流式响应（Streaming）**
  - 默认启用，通过 `stream: true` 参数请求 SSE 流式输出
  - 流式响应允许边生成边显示，大幅改善"等待感"：首 token 到达后立即开始填入 SCM 输入框
  - 流式响应期间用户可见逐步生成的文本，体验类似 GitHub Copilot 的 ghost text
  - 支持配置项 `aicommit.enableStreaming` 控制开关（默认 true）
  - 流式响应失败时自动降级为非流式请求
- 🆕 **HTTP 代理支持**
  - 支持 `aicommit.proxy` 配置项，格式：`http://host:port` 或 `https://host:port`
  - 支持 SOCKS5 代理：`socks5://host:port`
  - 代理配置也可通过环境变量 `HTTP_PROXY` / `HTTPS_PROXY` 读取
  - 代理认证：支持 `http://user:pass@host:port` 格式

### 5.4 性能优化
- **Diff 变更复用**：对 diff 内容做 hash 比对，仅当 diff 未变化时复用上次生成结果，避免重复调用 AI
- 增量分析代码变更
- 异步处理 AI 请求
- 避免阻塞主线程
- Diff 截断策略：当 diff 超过阈值时，优先保留变更摘要信息（文件名 + 变更类型 + 行数统计），截断具体代码内容

### 5.5 错误处理
- 完善的错误提示和日志
- 网络错误处理和重试机制
- AI 模型响应错误处理
- Git 操作错误处理
- 错误分类与用户提示：
  - 网络错误 → "网络连接失败，请检查网络设置"
  - API 认证失败 → "API 密钥无效，请检查配置"
  - API 额度不足 → "API 额度不足，请检查账户余额"
  - AI 响应格式异常 → "AI 返回内容无法解析，已填入原始文本"
  - Git 未安装 → "未检测到 Git，请先安装 Git"
  - 暂存区为空 → "暂存区无变更，请先暂存需要提交的文件"
  - 代理连接失败 → "代理连接失败，请检查代理配置"
- 🆕 **AI 响应后处理管线**：AI 原始输出需经过以下清理步骤后才填入输入框，确保结果可用：
  1. **Markdown 清理**：移除 AI 可能添加的 ` ``` ` 代码块标记、`**` 加粗标记等
  2. **多候选解析**：按 `---CANDIDATE N---` / `---END---` 分隔符拆分候选方案，提取第一条填入
  3. **XML 提取**：对 Semantic 模板，从 `<commit>` 标签中提取最终 commit 信息
  4. **行长度截断**：标题行超出 `maxLength` 配置（默认 72 字符）时截断并加 `...`
  5. **首行格式校验**：若首行不匹配任何已知格式模式（Conventional / Bullet / 自定义），标记为"格式异常"走降级流程
  6. **尾随空白清理**：移除多余空行和行尾空格
  - 后处理管线为可插拔设计，每个步骤为独立函数，便于单独测试和扩展

### 5.6 Prompt 工程

Prompt 设计是决定 commit 信息质量的核心环节，需明确定义以下内容：

#### 5.6.1 Prompt 模板结构
```
[System Prompt]
你是一个专业的 Git Commit 信息生成助手。请根据提供的代码变更，生成符合 {format} 规范的 commit 信息。
要求：
- 准确概括变更的目的和内容
- 使用{language}撰写
- 遵循指定的 commit 格式规范
- 生成 {candidateCount} 条不同风格的候选方案
- 使用 ---CANDIDATE N--- 分隔每个候选方案，以 ---END--- 结束

[User Prompt]
项目名称: {projectName}
变更文件数: {fileCount}
变更统计: +{additions} -{deletions}

代码变更内容:
{diffContent}
```

其中 `{language}` 由 `aicommit.commitLanguage` 配置项决定，取值为 "English"（默认）或 "中文"，确保 AI 输出使用指定语言。

#### 5.6.2 Token 预算分配
- 总预算：根据模型 `max_tokens` 配置，默认按如下比例分配：
  - Prompt（系统 + 用户）：不超过总 Token 的 70%
  - 输出预留：至少 30%
- Diff 内容在 Prompt 中的占比上限：60%
- 超出预算时的截断优先级：具体代码行 > 行级变更摘要 > 文件级摘要

#### 5.6.3 不同 Commit 格式的 Prompt 差异
- **Conventional Commits**：标准格式，标题行 `type(scope): description`，空行后接正文段落，正文详细描述变更内容
- **Gitmoji + Conventional**：在 Conventional 基础上，标题行前增加语义化 Emoji 前缀，如 `✨ feat(auth): 添加用户登录功能`
- **Bullet 要点式**（AI Commit 内置风格）：标题行末尾加冒号 `type(scope): description:`，正文每行以 `- ` 开头描述一个变更点，中间行以 `;` 结尾，最后一行以 `。` 结尾，标题与正文间无空行
- **自定义格式**：将用户定义的格式模板注入 Prompt，要求 AI 严格遵循

#### 5.6.4 项目上下文增强
- 可选将最近的 commit 历史（最近 5 条）作为上下文加入 Prompt，帮助 AI 理解项目的提交风格
- 项目名称自动从 git remote URL 提取，加入 Prompt 上下文

#### 5.6.5 Prompt 模板系统

支持用户自定义和切换 Prompt 模板，提供默认模板和多个备选模板。

**多候选输出格式约定**：所有模板在生成多候选方案时，要求 AI 在末尾使用统一的分隔格式，便于程序解析：
```
---CANDIDATE 1---
<commit message 1>
---CANDIDATE 2---
<commit message 2>
---END---
```

**默认模板 — Conventional Expert**
```
你是一个专业的 Git Commit 信息生成助手。请根据提供的代码变更，生成符合 Conventional Commits 规范的 commit 信息。
要求：
- 严格遵循 type(scope): description 格式
- type 只能是: feat, fix, docs, style, refactor, perf, test, chore, ci, build, revert
- scope 为可选，表示变更影响范围
- description 使用祈使句，首字母小写，不加句号
- 标题行与正文之间空一行
- 正文每条以 "- " 开头，说明具体变更内容
- 如需生成多条候选，使用 ---CANDIDATE N--- 分隔，以 ---END--- 结束
```

**备选模板 1 — Bullet（要点式风格）**
```
你是一个专业的 Git Commit 信息生成助手。请根据提供的代码变更，生成要点式的 commit 信息。
格式要求：
- 标题行: type(scope): description:（末尾加冒号）
- 正文紧跟标题行，中间不加空行
- 正文每行以 "- " 开头描述一个变更点
- 中间行以 ";" 结尾，最后一行以 "。" 结尾
示例:
feat(auth): 添加用户登录功能:
- 实现登录表单验证;
- 添加 JWT 令牌生成;
- 集成 OAuth 第三方登录。
如需生成多条候选，使用 ---CANDIDATE N--- 分隔，以 ---END--- 结束
```

**备选模板 2 — Concise（简洁风格）**
```
分析 git diff 并生成简洁的 commit message。只生成标题行，不生成正文。
格式: type(scope): description
示例: "refactor(auth): 简化 token 刷新逻辑"
如无法确定 scope 则省略。优先使用最常见的 type。
如需生成多条候选，使用 ---CANDIDATE N--- 分隔，以 ---END--- 结束
```

**备选模板 3 — Detailed（详细风格）**
```
你是一位资深开发者，正在为团队编写详细的 commit 信息。
要求：
- 标题行: type(scope): description
- 标题行与正文之间空一行
- 正文部分需要详细说明：
  1. 为什么要做这个变更（动机）
  2. 具体做了什么（实现方式）
  3. 可能的影响范围（风险评估）
- 正文每条以 "- " 开头，独立成行
如需生成多条候选，使用 ---CANDIDATE N--- 分隔，以 ---END--- 结束
```

**备选模板 4 — Semantic（语义化风格）**
```
你是一个 commit 信息生成专家。请使用结构化 XML 格式分析变更：

<analysis>
- 分析变更类型（feature/fix/refactor/docs/style/test/chore）
- 识别影响范围（scope）
- 评估变更重要性（major/minor/patch）
</analysis>

<commit>
type(scope): description

- 变更说明1
- 变更说明2
</commit>

要求基于实际代码变更生成，不要臆测。分析过程在 <analysis> 中完成，最终 commit 信息在 <commit> 中输出。
如需生成多条候选，每个候选用 ---CANDIDATE N--- 分隔，包含完整的 <analysis> 和 <commit>，以 ---END--- 结束
```

**备选模板 5 — Team（团队协作风格）**
```
为团队项目生成 commit 信息，需要让其他成员快速理解变更内容。
格式要求：
- 标题行: type(scope): description（不超过 72 字符）
- 标题行与正文之间空一行
- 正文每条说明一个变更点，以 "- " 开头
- 如果是破坏性变更，在 type 后加 "!": type(scope)!: description
- 如果关联 Issue，在正文末尾另起一行添加: Refs #issue_number
如需生成多条候选，使用 ---CANDIDATE N--- 分隔，以 ---END--- 结束
```

**自定义模板**
- 用户可在配置面板中编辑自定义 Prompt
- 支持占位符：`{format}`, `{candidateCount}`, `{maxLength}`, `{projectName}`, `{diffContent}`, `{fileCount}`, `{additions}`, `{deletions}`, `{language}`
- 占位符在渲染时自动替换为实际值，未匹配的占位符保留原样
- 提供模板编辑器，带语法高亮和占位符自动补全
- 支持导入/导出模板（JSON 格式）
- 模板校验：保存时检查占位符语法和必填字段是否完整

### 5.7 Token 用量统计

### 5.7.1 功能概述
每次 AI 生成 commit 信息后，统计并展示本次请求的 Token 用量和预估费用，帮助用户了解 AI 调用成本。

### 5.7.2 Token 计算方案

**方案 A：使用 API 响应中的 usage 字段（推荐）**
- OpenAI 兼容 API 的响应中包含 `usage` 字段：
  ```json
  {
    "usage": {
      "prompt_tokens": 1523,
      "completion_tokens": 87,
      "total_tokens": 1610
    }
  }
  ```
- 优点：精确，无需额外依赖
- 缺点：部分 API 提供商可能不返回此字段

**方案 B：使用 js-tiktoken 本地计算**
- 安装 `js-tiktoken` 包（纯 JS 实现，无需 WASM）
- 在发送请求前计算 prompt_tokens，收到响应后计算 completion_tokens
- 优点：不依赖 API 响应格式，可离线计算
- 缺点：增加包体积（约 100KB），不同模型的 tokenizer 不同

**方案 C：混合方案（推荐实现）**
1. 优先使用 API 响应中的 `usage` 字段
2. 若 `usage` 字段缺失，使用 `js-tiktoken` 估算：
   - GPT-4o / GPT-4o-mini 使用 `o200k_base` 编码器
   - GPT-4 / GPT-3.5 系列使用 `cl100k_base` 编码器
   - 根据模型名称自动选择对应编码器，未知模型默认使用 `cl100k_base`
3. 若 `js-tiktoken` 不支持该模型，使用近似估算：中文约 1.5 字/Token，英文约 4 字符/Token

### 5.7.3 费用计算

维护一份主流模型的单价表（每 1K Tokens 的美元价格）：

| 模型 | 输入价格 ($/1K) | 输出价格 ($/1K) |
|------|----------------|----------------|
| gpt-4o | 0.0025 | 0.01 |
| gpt-4o-mini | 0.00015 | 0.0006 |
| gpt-4-turbo | 0.01 | 0.03 |
| gpt-3.5-turbo | 0.0005 | 0.0015 |
| deepseek-chat | 0.00014 | 0.00028 |
| deepseek-reasoner | 0.00055 | 0.00219 |
| claude-3.5-sonnet | 0.003 | 0.015 |
| qwen-plus | 0.0004 | 0.0012 |
| qwen-turbo | 0.00005 | 0.0002 |

费用计算公式：
```
费用 = (prompt_tokens / 1000 × 输入单价) + (completion_tokens / 1000 × 输出单价)
```

- 支持用户自定义模型单价（在配置面板中添加），自定义单价的优先级高于内置单价表
- **内置单价表随插件版本更新，但模型厂商调价可能存在延迟**，用户可在配置中覆盖
- **支持多货币显示**：默认显示美元，用户可在配置中切换为人民币（CNY），使用可配置的汇率进行换算（默认汇率 1 USD = 7.2 CNY，用户可自定义）

### 5.7.4 统计数据展示

**单次生成后展示**：
- 鼠标悬停在状态栏图标时，tooltip 显示：
  ```
  AI Commit - 上次生成统计
  ─────────────────────
  配置组: GPT-4o
  模型: gpt-4o
  Prompt Tokens: 1,523
  Completion Tokens: 87
  总 Tokens: 1,610
  预估费用: $0.0047 / ¥0.034
  ─────────────────────
  今日累计: 12,340 tokens / $0.038
  ```

**累计统计**：
- 按日/周/月统计 Token 用量和费用
- 数据存储在 `globalState` 中，按日期 key 存储
- 配置面板中增加「用量统计」卡片，展示图表

**预算告警**：
- 支持设置日/月费用预算上限（默认关闭）
- 当累计费用达到预算的 80% 时，弹出 Warning 通知
- 当累计费用达到预算的 100% 时，弹出 Error 通知并询问是否继续使用
- 预算告警为软限制，不强制阻断功能，仅提醒用户

### 5.7.5 数据存储与清理

**存储结构**：
```typescript
interface TokenUsageRecord {
  date: string;           // "2026-04-18"
  modelGroup: string;     // "GPT-4o" — 🆕 记录使用的配置组名称
  model: string;          // "gpt-4o"
  promptTokens: number;
  completionTokens: number;
  totalTokens: number;
  cost: number;           // 美元
  timestamp: number;      // Unix timestamp
}

// globalState key: "aicommit.tokenUsage"
// 值为 TokenUsageRecord[]，最多保留 90 天数据
```

**自动清理策略**：
- 每次插件启动时，检查并删除 90 天前的记录
- 清理操作在后台异步执行，不阻塞插件激活
- 清理前在 Output Channel 记录 DEBUG 级别日志

**数据导出**：
- 提供 `AI Commit: 导出用量数据` 命令，将统计数据导出为 JSON 或 CSV 文件
- CSV 格式可用 Excel 打开，方便团队进行预算核算和成本追踪
- JSON 格式保留完整结构化数据，适合程序化处理

- 提供 VSCode Output Channel（`AI Commit`），记录关键操作日志

## 7. 安全考虑

### 7.1 API 密钥安全
- 使用 VSCode 的 `SecretStorage API` 加密存储 API 密钥（而非明文存储在 settings.json 中）
- 避免在日志中记录 API 密钥
- 支持使用环境变量存储 API 密钥（`${env:API_KEY_NAME}` 格式）
- API 密钥在配置界面中以密码形式遮盖显示
- 🆕 **Settings Sync 兼容说明**：VSCode 的 Settings Sync 不会同步 SecretStorage 中的密钥。在新设备上首次使用时，用户需重新输入 API 密钥。插件在检测到密钥为空时，应主动提示用户配置，而非静默失败

### 7.2 数据隐私
- 只发送必要的代码变更信息到 AI 服务
- 提供 `.aicommitignore` 机制：用户可指定某些文件的 diff 不发送给 AI（如含敏感信息的配置文件）
- 提供数据处理选项（如本地处理或使用私有 AI 模型）
- 遵循 GDPR 和其他数据隐私法规

### 7.3 代码安全
- 避免执行恶意代码
- 验证用户输入
- 防止命令注入（git 操作参数需严格转义）

## 8. 可测试性

### 8.1 测试策略
AI 生成结果具有随机性，需采用分层测试策略：
- **Prompt 模板测试**：验证不同配置下 prompt 模板的正确拼接和 Token 计算是否准确
- **Diff 解析测试**：使用固定 diff 样本验证解析逻辑的准确性
- **API 适配层测试**：使用 Mock Server 验证请求格式和错误处理逻辑
- **端到端测试**：使用固定响应的 Mock AI，验证完整流程（diff → 生成 → 填入 → 提交）

### 8.2 可测试性设计
- AI 调用层与业务逻辑解耦，支持注入 Mock 实现
- Diff 解析器为纯函数，易于单元测试
- 配置读取通过接口抽象，测试时可注入预设值

  - ✅ 预算告警：日/月预算设置，80% Warning + 100% Error 通知
  - ✅ 多货币显示：支持 USD/CNY，可配置汇率
  - ✅ 配置面板用量统计卡片：今日/本周/本月统计 + 14 天趋势图
- **Prompt 模板系统**（✅ 已实现，详见 5.6.5 节）
  - 6 个内置模板（Conventional Expert / Bullet / Concise / Detailed / Semantic / Team） + 自定义模板
  - 自定义模板支持占位符自动替换
  - 通过 `aicommit.promptTemplate` 配置项切换
- **`.aicommitignore` 敏感文件过滤**（✅ 已实现）
  - 支持类 `.gitignore` 语法（`*`、`**`、`?`、`!` 取反、`#` 注释）
  - 在 `getStagedDiff()` 中自动加载并过滤匹配文件的 diff
  - 过滤结果记录到日志
- **生成历史记录与回溯**（✅ 已实现）
  - 通过 `aicommit.saveHistory` 配置项控制是否保存
  - 历史存储在 `globalState`，最多保留 100 条
  - `AI Commit: 使用历史记录` 命令支持搜索、选择并填入
  - `AI Commit: 清空生成历史` 命令支持一键清空
- **诊断日志导出**（✅ 已实现）
  - `AI Commit: 导出诊断日志` 命令导出当前会话日志为 .txt 文件
  - Logger 内置缓冲区（最多 5000 行），支持 `getLogContent()` 读取
- ✅ **流式响应（Streaming）**：配置项已实现（`aicommit.enableStreaming`），SSE 流式解析待后续版本实现（当前为同步等待模式）
- ✅ **AI 响应后处理管线**：6 步清理管线（Markdown 清理 → 多候选解析 → XML 提取 → 行长度截断 → 格式校验 → 尾随空白清理），可插拔设计（`ResponsePipeline` 类）
- ✅ **Commit 语言配置**：`aicommit.commitLanguage` 支持 English / Chinese / Follow VSCode
- ✅ **多工作区支持**：多根工作区下自动识别目标仓库，多仓库时自动查找有暂存变更的仓库（⚠️ 暂为自动选择，未实现 QuickPick 用户手动选择）
- ✅ **Diff 来源选择**：`aicommit.diffSource` 支持暂存区（staged）/ 工作区（unstaged）两种 diff 来源
- ✅ **AI 模型配置组**：`aicommit.modelGroups` + `aicommit.activeModelGroup`，支持多组配置、QuickPick 切换、状态栏显示当前组名、每组独立 API Key 存储在 SecretStorage、配置面板添加/切换/删除组
- ✅ **项目级配置文件**：`.aicommitrc.json` 已实现 `ProjectConfigLoader`，已集成到 `configManager` 配置流程（项目级 > 用户级 > 默认值），支持文件监听和自动重载
- ✅ **Settings Sync 兼容**：密钥为空时提示"Settings Sync 不会同步 SecretStorage 中的密钥，请重新输入"
- ✅ **TokenUsageRecord.modelGroup**：用量记录包含 `modelGroup` 字段，CSV 导出包含该列
- ✅ **状态栏 tooltip 显示配置组名称**：齿轮图标 tooltip 显示当前活跃配置组名
- ✅ **大 diff 处理三选项**：截断后提供"继续生成（使用截断内容）/ 仅分析部分文件 / 取消"三个选项
- ✅ **SSE 流式响应**：`enableStreaming` 配置项已完整实现，开启时请求发送 `stream: true` 并解析 SSE 事件流，逐步拼接 `delta.content`，支持 `stream_options.include_usage` 获取用量数据，关闭时使用同步等待模式
- ✅ **SOCKS5 代理**：`proxy` 配置项支持 `socks5://` / `socks5h://` / `socks:` 协议，原生 Node.js 实现 SOCKS5 CONNECT 隧道（含认证），同时支持 HTTP/HTTPS 代理，环境变量增加 `ALL_PROXY` 回退
- ❌ **独立预览窗口**：多候选详细对比暂用 QuickPick 替代
- ❌ **i18n 国际化**：所有 UI 文本硬编码中文，暂未实现

## 11. 结论

本需求文档详细描述了 VSCode AI Commit 插件的功能、界面设计和技术实现方案。通过提供与 Trae 相同的 AI 生成 commit 信息功能，并支持自定义 AI 大模型，该插件将为 VSCode 用户提供更加便捷、高效的 Git 提交体验。

插件的核心价值在于：
- 自动化生成高质量的 commit 信息
- 支持多种 AI 模型和配置选项
- 无缝集成到 VSCode 的 Git 工作流中
- 提供简洁直观的用户界面
- 确保性能和安全性
- 完善的边缘场景处理和错误恢复机制

通过本插件，用户可以在 VSCode 中享受到与 Trae 相同的 AI 辅助 commit 功能，提高代码提交的规范性和效率。
